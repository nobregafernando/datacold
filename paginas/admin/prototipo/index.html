Três modelos de instalação

A topologia muda com a escala — de uma prova de conceito (1 sensor) até a planta inteira com vários quadros distribuídos. Em todos, o protocolo entre sensor e logger é RS-485 / Modbus RTU; do logger pra cima é MQTT.

Equipamento

É o que você quer medir: um motor, uma câmara fria, uma porta. O sensor "lê" alguma grandeza dele (calor, corrente, abertura).

🌡️

Sensor

Transdutor físico. Converte a grandeza (°C, A, V, contato) em sinal elétrico digital.

Modbus RTU sobre RS-485 · até 32 dispositivos por barramento
📟

Logger

"Tradutor". Lê o sensor pelo cabo RS-485 e fala MQTT na rede. Tem RAM/Flash pra segurar dados se a internet cair (16-64 MB).

☁️

Nuvem

Recebe via MQTT, valida, grava no banco. Aqui já estamos fora da fábrica.

📱

Web / App

Consulta a API pra mostrar gráficos, alertas e o painel ao vivo.

  • 1 sensor → 1 logger via RS-485 Modbus RTU
  • Logger sobe direto pra nuvem (sem broker / sem switch local)
  • Se a internet cair, o logger guarda em memória interna (16-64 MB) e re-envia. Sem fila externa = capacidade limitada.
Quando usar: teste inicial, validação do canal sensor→nuvem, 1 equipamento isolado.

Vários sensores

Temperatura, energia, porta — tudo no mesmo barramento RS-485. Cada sensor tem um endereço Modbus único (1, 2, 3…).

📟

Logger central

Faz polling sequencial: pergunta pro sensor 1, depois 2, depois 3… Empacota tudo e publica como MQTT.

🔀

Switch + APs Wi-Fi

Switch interconecta logger ↔ servidor. APs adicionam redundância — se um cair, o outro mantém a comunicação.

🖥️

Servidor / Broker local

Roda Mosquitto/EMQX. Tem fila persistente em disco: se a internet cair, segura tudo e re-envia depois.

É o ponto-chave da resiliência
☁️

Nuvem + Banco

Consolida dados de longo prazo, expõe API REST. Painel web e app consomem daqui.

  • Vários sensores num mesmo ponto físico (temp + energia + porta)
  • Conectados ao mesmo barramento RS-485 · cada um com endereço Modbus diferente
  • 1 logger → switch → APs Wi-Fi → servidor local com broker MQTT + fila
  • Resiliência total: se a internet cair, zero perda de dados (fila no disco do servidor)
Quando usar: 1 equipamento crítico ou 1 câmara fria com vários sensores próximos. É o que está rodando neste protótipo.

Quadros locais

Cada ambiente vira uma "ilha" auto-suficiente: sensores + logger + switch. Falha em 1 quadro não afeta os outros.

📻

LoRa (opcional)

Sensor sem fio de longa distância: 2 a 10 km, baterias que duram anos. Ideal pra sensores externos (silos, pátios, depósitos longe da fábrica) onde cabear Ethernet seria caro/inviável.

Encaixa no Modelo 3 quando há sensor remoto isolado
🏢

Centro da fábrica

Ponto único onde tudo converge: broker MQTT + fila em disco + gateway LoRaWAN. Os 3 quadros (e qualquer sensor LoRa remoto) entram aqui.

Mesma resiliência do Modelo 2: se a internet cair, segura a fila e re-envia.
📱

Visão global

O painel vê todos os ambientes simultaneamente, alertas cruzados (porta da câmara aberta + temperatura subindo) e correlações.

  • Cada ambiente (extrusão, câmaras frias, graxaria, área externa…) tem seu próprio quadro
  • Quadros próximos: Ethernet/Wi-Fi industrial. Quadros remotos: LoRa via gateway LoRaWAN
  • Tudo converge no mesmo broker MQTT central da fábrica
  • Falha em 1 quadro não afeta os outros (isolamento físico e elétrico)
  • Escala pra dezenas de pontos sem cabeamento de chão longo
Quando usar: planta inteira, vários ambientes — caso real Dale Sorvetes: 14 sensores em 6 ambientes, incluindo dois externos (Campo Grande e Três Lagoas).

Onde mora a reconexão automática?

A regra de ouro: quem segura o dado é o ponto mais próximo do sensor que tem memória e fila. Sempre dois pontos: logger e servidor/gateway local.

✓ FAZ

Logger

Recebe leituras dos sensores e tenta entregar pro broker MQTT. Se falhar, guarda na fila local e tenta de novo com backoff exponencial (5s → 15s → 30s → 1min).

É a primeira linha de defesa contra perda de rede local.

✓ FAZ

Servidor / Gateway local

Mantém duas reconexões: uma com os loggers (detecta logger mudo) e outra com a nuvem (segura dados em fila enquanto a internet estiver fora e sincroniza quando voltar).

É o ponto mais importante — garante histórico completo na nuvem.

~ ÀS VEZES

Sensor inteligente

Só se for sensor com Wi-Fi/4G e firmware próprio (raro em câmara fria industrial). Sensor "burro" só mede — não sabe que a internet caiu.

✗ NÃO FAZ

Switch

Switch só comuta pacotes. Não sabe que aquele tráfego é leitura de temperatura, não tem buffer de aplicação, não reenvia nada.

Sequência típica de retry no logger:
tentar envio → falhou → guarda na fila → espera 5s → tenta de novo → conexão voltou → drena a fila em ordem

O intervalo cresce a cada falha (backoff exponencial) pra não martelar a rede caída. Quando reconecta, envia tudo que ficou represado, na ordem original.

Como funciona a fila / buffer local

É o que separa "tem MQTT" de "tem MQTT à prova de queda de internet". A fila persiste em disco; o backup duplo (HD1 + HD2) protege contra falha física.

1

Sensor entrega

Logger recebe a leitura crua pelo barramento (Modbus, 4-20mA, pulso).

2

Publica no broker

Logger publica via MQTT com QoS 1 ou 2 — exige confirmação do broker antes de marcar como "entregue".

3

Broker enfileira

Servidor local guarda a mensagem em disco persistente (não só RAM). Sobrevive a reboot.

4

Replica em HD2

Cada mensagem é gravada em dois HDs em paralelo. Falha de disco não derruba o histórico.

5

Sincroniza na nuvem

Worker independente lê da fila e envia pro banco central. Marca como sincronizado só depois da confirmação 200 OK.

6

Limpa a fila

Mensagens confirmadas saem da fila local após X dias (retenção configurável — padrão 30 dias).

Sem fila local
  • Internet cai → leituras se perdem em RAM
  • Reboot do servidor → dado vai pro lixo
  • Auditoria fica com furos
Com fila local + backup duplo
  • Internet pode cair por horas — nada perdido
  • Servidor reinicia — fila persistente recupera tudo
  • Disco queima — HD2 mantém histórico

Conceito-chave: a fila local só é "esvaziada" depois que o destino final (banco na nuvem) confirma o recebimento. Isso garante entrega exatamente uma vez mesmo com retries.
